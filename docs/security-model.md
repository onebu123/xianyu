# 第2轮安全与权限体系说明

## 目标

本轮解决的是“私有部署后台不能裸奔”这个最小安全边界问题，覆盖角色隔离、接口权限、审计日志、敏感配置加密、Token 续期和基础限流。

## 角色模型

- `admin`：系统管理员。可管理账号、敏感配置、审计日志、店铺接入、提现审核和所有业务模块。
- `operator`：运营。可管理店铺、商品、订单、卡密、分销和常用运营模块，不可访问账号管理、审计日志、敏感配置和资金审核。
- `support`：客服。可查看统计、订单、售后与客户相关页面，不可管理店铺、资金和系统安全配置。
- `finance`：财务。可查看报表和资金中心，并执行提现审核，不可管理店铺、账号和敏感配置。

## 权限边界

- 后端接口按角色做强校验，前端菜单和按钮只做辅助收敛，不作为唯一安全手段。
- `system-accounts`、`open-logs`、`system-configs` 仅 `admin` 可访问。
- 店铺接入、授权、激活仅 `admin` 和 `operator` 可执行。
- 提现审核仅 `admin` 和 `finance` 可执行。
- 订单 CSV 导出仅 `admin`、`operator`、`finance` 可执行。

## 登录与 Token 续期

- 登录成功后返回 `token` 和 `expiresAt`。
- 前端在过期前 5 分钟自动调用 `/api/auth/refresh` 续期。
- 若本地只残留旧 Token 而没有 `expiresAt`，前端会在初始化阶段先取用户资料，再主动刷新一次令牌。
- 停用账号即使拿着旧 Token，也会在后端校验阶段被拦截。

## 审计日志

以下行为会写入 `audit_logs`：

- 登录成功
- 登录失败
- 未授权访问
- 高风险写操作触发限流
- 店铺授权与激活
- 订单导出
- 提现审核
- 账号创建、角色变更、状态变更
- 敏感配置更新

每条审计记录至少包含：

- 操作人
- 时间
- 动作类型
- 目标对象
- 结果
- 来源 IP
- 脱敏后的说明

## 敏感配置加密

- 敏感配置统一保存到 `secure_settings`。
- 数据库存储字段为 `value_encrypted` 和 `value_masked`。
- 页面和接口只返回 `value_masked`，不返回明文。
- 审计日志只记录脱敏值，不记录明文。
- 加密密钥默认读取 `APP_CONFIG_CIPHER_SECRET`，未配置时回退到 `JWT_SECRET`，生产环境必须显式设置独立密钥。

## 基础限流

- 登录接口按 IP 限流。
- 账号管理、敏感配置更新、提现审核、店铺授权等高风险写操作按用户限流。
- 触发限流时会返回 `429`，同时写入审计日志。

## 验收建议

- 分别使用 `admin / operator / support / finance` 四类账号登录，确认菜单和按钮差异。
- 用 `operator` 访问 `system-accounts`，确认接口返回 `403`，并在审计页看到拒绝记录。
- 更新一条敏感配置后，检查页面只显示掩码值，数据库中不存在明文。
- 停用一个新建账号，再用该账号登录，确认被拒绝并产生审计日志。
