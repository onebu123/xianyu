# 回滚说明

## 适用范围

本文档用于第1轮交付范围内的私有化回滚流程，适用于单机 Docker Compose 部署。

## 回滚触发条件

满足以下任一情况，应立即考虑回滚：

- 服务无法启动
- 健康检查持续失败
- 管理员无法登录
- 关键页面白屏或接口大面积报错
- 升级后数据异常

## 回滚前确认

- 已保留升级前代码版本或镜像版本
- 已保留升级前 `.env` 备份
- 已保留升级前数据库备份

## 快速回滚步骤

### 1. 停止当前服务

```bash
docker compose down
```

### 2. 恢复代码版本

如果是 Git 部署：

```bash
git log --oneline -n 5
git checkout <上一个稳定版本提交>
```

如果是压缩包部署：

- 恢复到升级前的代码目录备份

### 3. 恢复配置

```bash
cp .env.bak.<时间戳> .env
```

### 4. 恢复数据库

```bash
cp server/data/app.db.bak.<时间戳> server/data/app.db
```

如有上传文件备份，也需要恢复。

### 5. 重启旧版本

```bash
docker compose up -d --build
```

### 6. 回滚后验收

```bash
docker compose ps
docker compose logs --tail=100
curl http://127.0.0.1:4300/api/health
```

## 回滚验收标准

- 健康检查恢复正常
- 运行模式与回滚前一致
- 管理员可以登录
- 关键页面可访问
- 数据恢复到回滚前预期状态

## 数据回滚注意事项

- 如果升级期间已经写入新业务数据，数据库回滚会丢失这些数据。
- 因此生产升级前必须安排维护窗口，避免业务写入与数据库回滚冲突。

## 最低记录要求

每次回滚后必须记录：

- 回滚时间
- 回滚原因
- 回滚前版本
- 回滚后版本
- 是否恢复数据库
- 是否恢复上传目录
- 后续修复计划
